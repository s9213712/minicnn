# MiniCNN Agent 指令檔（V1–V4 整合版）
# 用途：直接餵給 agent 執行
# 範圍：Template/Dataset 修復、CLI/Smoke/Dependency 修正、Windows/CUDA Build 文件對齊、ConvNeXt 能力導入
# 原則：先穩定，再擴充；先驗證，再開發；不破壞既有 CLI 契約

==================================================
0. 全域規則（強制遵守）
==================================================

你正在修改的專案是一個正在往 framework 化演進的 MiniCNN 專案。
本次任務分成兩大部分：
1) V1–V3：穩定化與 onboarding 修正
2) V4：ConvNeXt capability planning 的最小落地版本

在任何修改前，遵守以下規則：

[Rule A] 不可破壞現有 CLI 名稱與主要入口，除非保留相容 alias。
[Rule B] 不可用 silent fallback 偽裝 backend 支援能力。
[Rule C] Optional dependency 缺失不可當成 repo 核心損壞。
[Rule D] Template 視為半公開 API，必須可執行，不可只是示例。
[Rule E] 文件中的指令、路徑、輸出結果必須與實作一致。
[Rule F] 每個使用者可見的錯誤都必須提供 Cause / Fix / Example。
[Rule G] 在進入 ConvNeXt 開發前，必須先通過穩定化驗收。

3. Phase 1 — Template & Dataset Hardening
==================================================

目標：讓 templates 真正可跑，不再預設爆掉。

3.1 修復 MNIST templates
------------------------------------------
請檢查並修正：
- templates/mnist/lenet_like.yaml
- templates/mnist/mlp.yaml

強制條件：
- num_samples + val_samples <= 60000

建議預設值：
- num_samples: 55000
- val_samples: 5000

要求：
- 不要只修數字，還要同步確認 loader 與文件語意一致。
- 若 config schema 還有其他容易踩坑的欄位，也一併修正預設。

3.2 新增 template validity 測試
------------------------------------------
新增或擴充測試，例如：
- tests/test_templates_validity.py

測試至少覆蓋：
- templates/**/*.yaml 可 parse
- dataset split 合法
- 常見 template 不會在初始化階段就拋 ValueError
- 對 MNIST 與 CIFAR templates 分別做基本 sanity check

3.3 文件化 dataset split 規則
------------------------------------------
更新：
- templates/README.md
- docs/USAGE.md
- 任何新手入口文件（如 README 或 quickstart 區塊）

必須清楚說明：
- MNIST 的 60000 是 train pool
- validation 是從 train pool 切出，不是額外資料
- CIFAR 的 train / val / test 來源與切分語意
- num_samples / val_samples 的實際意義

3.4 防呆
------------------------------------------
若使用者配置超過可用樣本量，錯誤必須長這樣：

[ERROR] Dataset split invalid
-> Cause: num_samples + val_samples exceeds available training samples
-> Fix: reduce num_samples or val_samples
-> Example:
   num_samples=55000
   val_samples=5000

==================================================
4. Phase 2 — CLI / Smoke / Dependency 修正
==================================================

目標：讓 smoke/CLI 對 optional dependency 更誠實，避免誤判 repo 壞掉。

4.1 修正 develop 安裝策略
------------------------------------------
檢查 Makefile 的 develop 目標。

若目前 `make develop` 只安裝 .[dev]，但 smoke / flex registry 需要 torch 才能工作，
請改為更合理的預設，例如：
- python -m pip install -e .[torch,dev]

如果你認為不應強制安裝 torch，也可以保留 .[dev]，
但必須同時完成 4.2 / 4.3 / 4.4，確保 UX 不會誤導。

4.2 修正 smoke 行為
------------------------------------------
修改：
- src/minicnn/smoke.py

要求：
- torch 未安裝時：給 warning，不可把整個 repo 視為 fail
- flex registry 為空時：先檢查是否是因為 torch optional dependency 缺失
- native artifact 缺失時：可以是 error，但訊息要明確
- JSON 模式與文字模式都要一致

新增 smoke JSON 欄位，至少包含：
- torch_available
- cuda_available（若可判定）
- native_available
- flex_registry_ready（若適用）
- warnings / errors 分類

4.3 改善 CLI 錯誤訊息格式
------------------------------------------
所有使用者可見錯誤，統一成以下格式：

[ERROR] <問題摘要>
-> Cause: <直接原因>
-> Fix: <如何修>
-> Example:
   <一組可用配置或指令>

至少覆蓋：
- dataset split invalid
- torch missing
- native DLL / shared library missing
- backend capability not supported
- invalid config key / invalid component name

4.4 新增 smoke regression tests
------------------------------------------
新增測試以覆蓋：
- 無 torch 時 minicnn smoke 不 crash
- registry 缺失但 root cause 是 optional dependency 時，不回報誤導性錯誤
- JSON payload 正確標示 dependency 狀態

4.5 拆分錯誤類型（若現況混亂）
------------------------------------------
若目前 smoke / CLI 將所有問題都混成 generic RuntimeError，
請做適度整理，至少在使用者輸出層分成：
- optional dependency issue
- user config issue
- native artifact issue
- internal repo wiring issue

5. Phase 3 — Windows / CUDA Build 文件與腳本對齊
==================================================

目標：文件與 PowerShell 腳本對齊，讓 Windows 使用者不再去錯地方找 DLL。

5.1 檢查文件與實際輸出路徑差異
------------------------------------------
檢查：
- docs/guide_windows_build.md
- scripts/build_windows_native.ps1

若文件仍假設 DLL 在：
- cpp\Release\*.dll

但腳本實際輸出在：
- cpp/build-windows-*/Release/*.dll
或其他 build dir

請以腳本實際行為為準，全面修正文檔。

5.2 強化 build_windows_native.ps1 的成功輸出
------------------------------------------
在 build 成功後，明確輸出：
- cublas DLL 完整路徑
- handmade DLL 完整路徑
- 下一步驗證指令
- 常見故障排查提示

示例格式：

Build SUCCESS

DLL location:
  cublas:    <full or repo-relative path>
  handmade:  <full or repo-relative path>

Next step:
  minicnn train-native ...

Troubleshooting:
  - If DLL not found -> verify CUDA toolkit / generator / build config
  - If load fails -> verify PATH and runtime dependencies

5.3 GPU 架構提示（建議做）
------------------------------------------
若腳本能檢測使用者 GPU 世代或使用者指定的 CudaArch，
請在不過度複雜的前提下，給出建議提示，例如：
- RTX 20xx -> 75
- RTX 30xx -> 86
- RTX 40xx -> 89
- RTX 50xx -> 可能依工具鏈支援情況提示使用對應 arch

若無法可靠偵測，也至少在文檔加上表格說明。

5.4 Windows 文件重構
------------------------------------------
在 docs/guide_windows_build.md 中補充：
- 需要哪些軟體
- Visual Studio / Build Tools 需求
- CMake / CUDA toolkit / Python 版本要求
- 哪些參數是不同 GPU 世代需要調整的
- 為什麼不要在 makefile 或腳本裡硬編某一版 VS generator
- 成功 build 後要去哪裡找 artifacts

6. Phase 4 — ConvNeXt Capability 最小落地
==================================================

目標：不是一次做完整現代框架，而是導入支撐 ConvNeXt 類模型的最小能力集。

在開始前，必須通過：
- minicnn smoke
- pytest tests/

若未通過，禁止進入本 Phase。

6.1 定義最小能力集
------------------------------------------
MiniCNN 必須能表達以下能力：
- depthwise conv
- pointwise conv (1x1)
- layernorm2d（channel-first / NCHW）
- GELU
- residual add
- stage stacking / block 組裝

請不要過度擴張，不要一口氣引入完整大型模型 zoo。
本 Phase 只做支撐 ConvNeXt-tiny 風格模型的最小抽象。

6.2 新增 layers / blocks / model
------------------------------------------
建議新增或整理模組，例如：
- src/minicnn/nn/layers/depthwise_conv2d.py
- src/minicnn/nn/layers/pointwise_conv2d.py
- src/minicnn/nn/layers/layernorm2d.py
- src/minicnn/nn/layers/gelu.py
- src/minicnn/nn/blocks/convnext_block.py
- src/minicnn/models/convnext.py

若專案現有命名風格不同，可遵循現有風格，但要保持一致。

6.3 Depthwise conv
------------------------------------------
要求：
- groups = in_channels
- Torch backend 可直接使用原生支援
- 若其他 backend 不支援，不可 silent fallback 成普通 conv
- 必須透過 capability system 或顯式錯誤處理

6.4 Pointwise conv
------------------------------------------
要求：
- 用 1x1 conv 表示
- 與 depthwise conv 可被 block 組合
- shape 檢查與錯誤處理要清楚

6.5 LayerNorm2d
------------------------------------------
要求：
- 支援 NCHW
- 不要偷偷要求 NHWC
- 文件與程式命名要講清楚這是 channel-first 的實作
- 若使用 torch backend，請確認其語意與你包裝層一致

6.6 GELU
------------------------------------------
要求：
- 使用可靠實作（torch 原生或標準近似式）
- 不要用錯 activation 名稱冒充 GELU

6.7 ConvNeXt block
------------------------------------------
block 最小結構要求：
x
-> depthwise conv
-> layernorm
-> pointwise conv expand
-> GELU
-> pointwise conv shrink
-> residual add

要求：
- shape 在 residual add 前後一致
- block 本身可獨立測試
- 若有 drop path / layer scale 等進階機制，先不要硬上，除非專案已有穩定抽象支撐

6.8 ConvNeXt model 組裝
------------------------------------------
新增最小可用 model，例如：
- convnext_tiny or convnext_like_small

要求：
- 能透過 config 指定，例如：
  model:
    name: convnext_tiny

- 能至少在 CIFAR-10 跑通 1 epoch
- 不要求這次就追求最終精度，但前向、反向、訓練循環要完整

6.9 Registry 擴充
------------------------------------------
更新 registry，例如 flex/components.py 或對應註冊點，加入：
- depthwise_conv2d
- pointwise_conv2d
- layernorm2d
- gelu
- convnext_block
- convnext model entry

要求：
- registry 缺失要報明確錯誤
- smoke 若有 registry surface 檢查，需同步更新

6.10 Backend capability 宣告
------------------------------------------
新增或整理 capability 機制，例如：
- src/minicnn/backend_capability.py
或其他更合適位置

至少要能表達：
- supports_depthwise_conv
- supports_pointwise_conv
- supports_layernorm2d
- supports_gelu
- supports_residual_add
- supports_convnext_block

要求：
- train / config validation / runtime 應能引用這些能力
- 不支援時要明確報錯，不可隱式降級

6.11 Config validation
------------------------------------------
若專案已有 config validation 機制，擴充它。
若沒有，請至少做最小版本，讓以下錯誤能在較早階段被發現：
- 不存在的 layer 名稱
- backend 不支援該 layer / block
- model template 指向不存在的 registry entry

若成本合理，可新增：
- minicnn validate-config

但前提是不要破壞現有 CLI。
