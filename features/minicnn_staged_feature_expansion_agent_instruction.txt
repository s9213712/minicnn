MINICNN FEATURE EXPANSION MASTER INSTRUCTION (STAGED EXECUTION)
Repository target: current uploaded repo `minicnn-main(2)`
Goal: expand configurable training/model options without destabilizing the repo.
Audience: coding agent working directly on this repo.

=====================================================================
0. MISSION AND OPERATING RULES
=====================================================================
You are modifying the CURRENT repo structure, not an older minimal_cuda_cnn layout.
This repo already contains multiple execution paths:

1) flex / torch stack
   - registry-driven
   - already supports multiple activations/losses/optimizers/schedulers via PyTorch
2) autograd / pure-python stack
   - custom Tensor/autograd engine
   - partial support for losses and optimizers
   - needs expansion and cleanup
3) unified / cuda_legacy path
   - heavily constrained handwritten CUDA legacy training path
   - must remain conservative and highly validated
4) framework registry / backend capability layer
   - currently only partially populated

Critical rule: DO NOT try to make all backends support all options in one pass.
Critical rule: prefer correctness, compatibility, and validation over breadth.
Critical rule: implement features in stages and stop at each stage with green tests.
Critical rule: if a backend does not support a feature, fail early with a precise validation error.

Required engineering behavior:
- keep backward compatibility for existing configs where practical
- do not silently reinterpret old configs in surprising ways
- update tests alongside code
- update docs/config examples only after code is working
- avoid invasive refactors unless they reduce long-term complexity

Definition of success:
- the repo becomes a configurable mini-framework with staged support
- flex stays broad and stable
- autograd gains real feature breadth
- cuda_legacy remains explicit, narrow, and safe

=====================================================================
8. CONFIG DESIGN GUIDELINES
=====================================================================
Aim for config consistency, but do not break working configs unnecessarily.

Recommended style:
- activation as canonical type string
- optimizer.type with optimizer-specific fields nested under optimizer
- loss.type
- scheduler.enabled + scheduler.type + scheduler params
- model.init or model.initialization.type for init policy
- explicit layer types for pooling/norm/dropout when layer-list config is used

Avoid:
- too many aliases without tests
- backend-specific magic hidden behind generic names
- silently ignoring unsupported keys

If unsupported keys are present for a chosen backend, prefer warning or validation error depending on severity.
